# Notifications — 何时推 Telegram

默认假设：凯歌在电脑边。终端里的文字就是通知，不需要额外推送。
Telegram 只在我判断凯歌**不在电脑边**或**需要越过终端**的场景使用。

## 触发 Telegram（用 `send-message` skill）

- **`/loop` 或 `/schedule` 任务的结果**：这些任务设计上就是脱离终端的，完成时必须推送。
- **显式要求**："发给我"、"notify me"、"推一下"、"告诉我一声"——没指定渠道时默认 Telegram。
- **长耗时任务完成**：单次任务如果预期运行 >5 分钟（大型构建、批量处理、长测试套件），收尾时推一条简短摘要。
- **btc-watch 等明确走 Telegram 的 skill**：按各 skill 自己的规则。

## 不触发 Telegram

- **常规对话收尾**：凯歌发消息 → 我执行 → 我回复。结果在终端里，不再推 Telegram。
- **快速任务完成**：秒级/几十秒级任务，终端输出足够。
- **错误/权限请求**：由 `notify-send` hook 负责（已在 `settings.json`）。不要 Telegram 重复。
- **中间进度更新**：除非凯歌显式要求进度推送，否则不要骚扰。

## 内容准则

- 简短：Telegram 消息不是会话 transcript，是**摘要**。一句话能讲完就一句话。
- 包含结论：成功/失败、关键数字、下一步是否需要凯歌介入。
- 不包含：工具调用细节、思考过程、完整 diff。
- 超长（>几百字）用 `--file` 附件，不刷屏。

## 边界

如果不确定当前任务是否属于"脱离终端"场景，默认不推——凯歌可以手动说"发给我"。
过度推送比偶尔漏推更讨厌。
